# 需求文档生成

将模糊的需求想法，通过结构化对话澄清，生成符合公司标准的需求文档。让人和 agent 都明确知道要构建什么。

## 触发条件

**自动触发** — 当检测到以下任一情况时，建议用户进入 PRD 流程：
- 用户描述的需求缺少背景（为什么做）
- 缺少明确的产出结果（用户在哪里看到什么、能做什么）
- 缺少验收标准（怎么算做完）
- 需求存在歧义（可以有两种以上理解）
- 涉及多个功能点但没有梳理清楚边界

**手动触发** — 用户输入 `/prd`

**退出机制** — 用户随时可以说"不需要 PRD"退出流程。

## 流程总览

```
用户提出需求
    ↓ agent 判断信息不足，建议进入 PRD 流程
    ↓ 用户同意
判断粒度 → 和用户商量用完整模式还是精简模式
    ↓
阶段一：读代码/文档 → brainstorming 对话 → 清晰度 7 项全过
    ↓ 用户确认"理解正确"
阶段二：Writer 写 PRD → Reviewer 审 → 对抗直到 Reviewer 通过条件全满足
    ↓
阶段三：用户审核 → 确认 → 存入 docs/prd/YYYY-MM-DD-<名称>.md
```

## 模式选择

进入流程后，agent 先评估需求粒度，然后和用户商量：

**完整模式**（新功能、新模块、涉及多角色多流程的需求）：
- 完整 brainstorming 对话
- Writer + Reviewer 多轮对抗
- 输出完整 PRD

**精简模式**（改一个按钮、调一个流程、加一个字段等轻量需求）：
- 快速澄清（可能 2-3 个问题就满足 7 项）
- Writer + Reviewer 单轮快速审查
- 输出完整模板结构，但允许部分章节写"不适用"

建议话术：
> "这个需求看起来比较轻量，建议用精简模式（快速澄清 + 单轮审查）。你觉得可以吗？还是想走完整流程？"

用户说了算，agent 只是建议。

## 阶段一：需求澄清（Brainstorming）

### 准备工作

在提问之前，先做：
1. 读取项目代码结构和关键文件，了解现有系统
2. 读取 `docs/` 目录下的已有文档
3. 读取 `docs/prd/` 下的已有 PRD，了解项目上下文
4. 基于以上信息，让提问更有针对性

**如果是对已有项目的修改（而非全新项目），必须额外做：**
5. 读取需求涉及的相关代码模块，了解现有实现逻辑和数据结构
6. 询问用户是否需要查看数据库结构（表结构、字段、关联关系），如用户同意则读取相关 schema
7. 基于代码和数据库的实际情况，在澄清过程中判断需求的技术可行性——如果发现某个需求在现有架构/数据结构上难以实现或成本极高，必须在澄清阶段就告知用户，而不是等到 PRD 写完才发现做不了

**核心原则：不能脱离代码和数据库空谈需求。** 所有澄清出的需求必须建立在对现有系统真实了解的基础上，避免"需求很完美但技术上实现不了"的情况。

### 澄清方法

- **一次只问一个问题**，不要一次抛出多个问题
- **优先用选择题**，需要时才用开放题
- 在合适时机**提出 2-3 种方案**，说明取舍，给出推荐
- 围绕 CREA 框架 + 产出结果追问，缺哪块问哪块

### 澄清主线

**过去（As-Is）—— Cause**：
- 谁（哪些用户/角色）
- 在哪里（什么页面/场景/环节）
- 做了什么（当前行为）
- 遇到了什么问题（痛点）
- 为什么现在要解决

**期望（To-Be）—— Result + 产出结果**：
- 谁（哪些用户/角色）
- 在哪里（什么页面/入口）
- 能做什么（新操作/新流程）
- 看到什么结果（反馈/状态变化/数据展示）
- 怎么衡量做成了（可量化的关键结果）

### 清晰度标准（出口条件）

每轮问完后自检以下 7 项，**全部满足**才算澄清完成：

```
□ Cause：能说清"谁在什么场景下遇到了什么问题，为什么要解决"
□ Result：期望的关键结果可衡量、可验证
□ 产出结果：每个涉及的用户角色，在哪个页面/入口，能做什么操作，看到什么结果——逐条明确，无遗漏
□ Effect：对现有系统、业务流程、现有客户的影响和风险已识别
□ Action：关键任务清单可拆成开发工作项
□ 边界清晰：明确知道什么不做
□ 无歧义：需求描述能映射出唯一的测试用例
```

**不是问了多少轮，而是 7 项是否全过。** 有的需求 2 轮就清楚，有的需要 15 轮。

### 中间检查点

当已经澄清了 4-5 项时，给用户一个阶段性总结：

> "目前我理解的需求是这样：[总结]。还差 [XX] 没有澄清，我们继续。"

这样即使对话很长，关键信息已被固化在总结里。

### 澄清完成确认

7 项全过后，向用户做最终确认：

> "需求已经澄清完毕，我理解的是：[完整总结]。确认无误的话，我开始生成 PRD。"

用户确认后进入阶段二。

## 阶段二：双 Agent 对抗生成 PRD

启动两个 subagent，角色定义有明确差异：

### Writer Agent

**角色**：产品经理视角。
**职责**：根据阶段一的澄清结果，严格按照公司 PRD 模板写文档。
**要求**：
- 完全遵循下方的 PRD 模板格式，不增不减
- 内容来自澄清阶段确认的信息，不自行发挥
- 需求描述做到 Testable（无歧义，可映射唯一测试用例）和 Economy（综合成本最低方案）
- 复杂逻辑提供流程图，关键字段提供字段表
- 埋点需求严格遵循公司埋点规范
- 能填的元信息填上，填不了的标注"待填写"

### Reviewer Agent

**角色**：开发 + 测试视角。"我拿到这个 PRD 能不能直接干活？"
**职责**：逐条审查 PRD，找出所有会导致开发卡住、测试无法写用例、验收标准模糊的问题。
**加载**：每次审查前先读取 `docs/prd/.reviewer-rules.md`（如存在），将历史审查经验纳入审查标准。

**Reviewer 通过条件**（全部满足 = 收敛）：

```
□ Testable —— 每条需求描述可映射唯一测试用例，无"等""相关""适当"等模糊词
□ 产出结果 —— 每行都具体到 用户角色 + 页面/入口 + 操作 + 结果，无遗漏
□ 用户故事可走通 —— 每个 user case 按场景描述逐步走，流程完整无断点，能验证 PRD 覆盖了所有关键路径
□ 验收清单覆盖 —— 主流程验收清单覆盖了产出结果表和用户故事的所有关键路径
□ 业务影响 —— 每个识别出的影响都有对应的规避方案
□ 埋点规范 —— 事件名称用功能模块以_拼接，eventId 用驼峰规则，无空格无连字符
□ 无占位符 —— 不存在 TBD、TODO、待补充、待确认（"待填写"的元信息字段除外）
□ 完整性 —— CREA 四部分 + 产出结果 + 用户故事都有实质内容
□ 可读性 —— 复杂逻辑有流程图，关键字段有字段名，大段描述用图表简化
```

### 对抗流程

```
Writer 写初稿
    ↓
Reviewer 逐条审查，输出具体问题列表
    ↓ 有未通过项？
    ↓ 是 → Writer 修改 → Reviewer 再审 → 循环
    ↓ 否 → 通过条件全满足 → 进入阶段三
```

**收敛条件**：Reviewer 通过条件 8 项全满足。不是固定轮次，是真的没问题了才通过。

---

## 阶段三：用户审核

将生成的 PRD 存入 `docs/prd/YYYY-MM-DD-<需求名称>.md`，请用户审核：

> "PRD 已生成并保存到 `docs/prd/YYYY-MM-DD-xxx.md`，请审核。如有修改意见请告诉我。"

- 用户有修改意见 → 修改后再请用户确认
- 用户确认 → PRD 定稿，流程结束

### Reviewer 进化

用户审核时如果提出了修改意见，agent 判断其中是否有**通用价值**（不是只针对这一篇 PRD 的特殊修改，而是以后所有 PRD 都应该注意的点）。如果有，追加到 `docs/prd/.reviewer-rules.md` 中。

**注意**：只追加通用性的审查经验，不追加个例修改。

## PRD 模板

以下是公司标准 PRD 模板，Writer 必须严格遵循此结构输出：

````markdown
# 需求文档

| 字段 | 内容 |
|------|------|
| Tower 任务 | 待填写 |
| 产品 | {产品名称} |
| 创建时间 | {YYYY-MM-DD} |
| 需求状态 | 草稿 |
| 初审时间 | 待填写 |
| 终审时间 | 待填写 |
| UI | 待填写 |
| UI图链接 | 待填写 |
| 开发成员 | 待填写 |
| 开发完成时间 | 待填写 |
| 测试成员 | 待填写 |
| 测试完成时间 | 待填写 |
| 上线时间 | 待填写 |

| 文档版本 | 编辑时间 | 编辑内容 | 编辑人 |
|---------|---------|---------|-------|
| v1.0 | {YYYY-MM-DD} | 初始版本 | Claude + {用户名} |

## 需求简介

### Cause: 需求背景

为什么要做这个功能，哪些客户在什么场景下遇到了什么问题。

### Result: 关键结果

要达成哪些可衡量的关键结果：
- 业务指标：{具体指标，尽量量化，需要有预期}
- 客户覆盖：{哪些客户要正常使用什么功能}

### 产出结果

最终用户可感知的变化：

| 用户角色 | 在哪里（页面/入口） | 能做什么（操作） | 看到什么结果 |
|---------|-------------------|----------------|------------|
| {角色} | {具体页面/入口} | {具体操作} | {具体反馈/状态变化} |

### Effect: 业务影响

对现有业务将带来哪些影响和风险，以及规避方案：
- 对已有功能的兼容影响：{影响} → 规避：{方案}
- 对现有业务流程的影响：{影响} → 规避：{方案}
- 对现有客户的影响：{影响} → 规避：{方案}

### Action: 关键任务

要达成关键结果，要做的关键任务清单（含跨部门协作）：
- [ ] {任务1}
- [ ] {任务2}

## 产品需求

### 需求描述

{具体功能规格，要求：}
- Testable可测试：需求描述无歧义，逻辑性强，测试可映射出唯一的测试用例
- Economy具有经济性：在不影响达成关键结果的前提下，采用综合成本最低的设计方案
- 阅读难度低：复杂逻辑提供流程图；关键字段提供字段名；大段需求描述采用图表简化

### 用户故事

用具体场景验证需求是否完整可走通。每个故事覆盖一条关键路径，走不通说明 PRD 有缺失。

| 编号 | 作为（角色） | 我想要（操作） | 以便于（目的） |
|------|------------|--------------|--------------|
| US-001 | {角色} | {具体操作} | {期望达成的目的} |

**场景描述：**

**US-001：{故事标题}**
{一段完整的场景叙述：谁在什么情况下，做了什么操作，系统如何响应，最终看到什么结果。要具体到可以按步骤走通。}

验收标准：
- [ ] 当 {触发条件} 则 {期望结果}
- [ ] 当 {触发条件} 则 {期望结果}

> 验收标准统一使用"当...则..."（WHEN...THEN...）格式，明确触发条件和期望结果，消除歧义。

### 埋点需求

当产品需求对后台新增可见功能时，需对功能进行埋点定义：

| 事件名称 | eventId | 含义（触发时机） | 属性 |
|---------|---------|----------------|------|
| {功能模块_子功能} | {驼峰规则命名} | {具体上报时机} | {个性属性定义} |

埋点命名规范：
- 事件名称基于功能模块以 `_` 拼接
- eventId 为落库英文事件名称，功能层级以 `_` 拼接
- eventId 不可出现空格及 `-`，如遇空格或 `-` 则替换为驼峰规则（去除空格和连字符，单词连起，连起处首字母大写）

如本需求不涉及新增可见功能，此章节标注"不适用"。

### 主流程验收清单

产品对需求主流程的验收质量负责。主流程中任一节点跑不通，即认为不符合验收标准，不再继续验收，打回重新测试。

- [ ] 当 {用户执行某操作/系统达到某状态} 则 {期望的结果/行为}
- [ ] 当 {用户执行某操作/系统达到某状态} 则 {期望的结果/行为}

> 所有验收条件统一使用"当...则..."（WHEN...THEN...）格式。

### 开发边界规则

约束开发实现时的行为边界，避免踩坑：

**Always（始终要做）：**
- {如：提交前必须跑完所有相关测试}
- {如：遵循现有代码命名规范}

**Ask first（需先确认）：**
- {如：数据库 schema 变更需先与 DBA 确认}
- {如：删除或修改现有 API 字段需先评估影响范围}

**Never（绝不能做）：**
- {如：不可提交密钥或敏感配置到代码仓库}
- {如：不可跳过类型检查}

## 上线方案

{选择以下方案之一：}
1. 全量上线
2. 新功能灰度上线：{灰度方案：什么时间、对哪些客户生效、什么时间全量上线}
3. 已有功能优化灰度上线：{灰度方案}

灰度上线到全量上线时间不可超过一周。
````

---

## 埋点命名规范（Writer 和 Reviewer 共同遵循）

- 事件名称：基于功能模块，以 `_` 拼接层级。例：`订单管理_批量导出`
- eventId：英文，功能层级以 `_` 拼接，不可有空格和 `-`。遇空格或 `-` 用驼峰替换。例：`orderManagement_batchExport`
- 含义：定义具体上报时机，如"用户点击批量导出按钮时触发"
- 属性：如需上报个性属性，定义属性名和含义

## Reviewer 进化机制

### 进化文件位置

`docs/prd/.reviewer-rules.md`

### 初始创建

如果项目中不存在此文件，在首次运行 PRD skill 时自动创建初始版本。

### 追加规则

每次用户在阶段三审核 PRD 后给出修改意见时，agent 判断：
- 这个意见是**通用的**（以后所有 PRD 都应注意）→ 追加到 reviewer-rules
- 这个意见是**个例的**（只针对这篇 PRD 的特殊情况）→ 不追加

追加格式：
```markdown
- {审查规则} —— 来源：{YYYY-MM-DD} {需求名称}
```

### 使用方式

Reviewer agent 每次启动时读取此文件，将其中的规则纳入审查标准，与内置的 8 项通过条件一并执行。
